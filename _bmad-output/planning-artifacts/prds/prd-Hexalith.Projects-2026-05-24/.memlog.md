- (decision) Hexalith.Projects v1 is an internal platform module for Hexalith.Chatbot that provides a durable tenant-aware AI workspace boundary, not a standalone or generic project-management product, so conversations can resume with correct project context.
- (decision) The primary user is a Hexalith.Chatbot user working across multiple conversations and resources, because continuity should not require rebuilding project context in every session.
- (decision) Projects owns project metadata, setup, lifecycle, and resource associations while Conversations, Folders, and Memories remain systems of record for their payloads and authorization, preventing duplicated data and boundary bypass.
- (decision) All project reads, writes, links, resolution, and context assembly are tenant-scoped and fail closed when authorization or resource availability cannot be verified, because preventing cross-project and cross-tenant leakage is the core safety constraint.
- (decision) A Conversation belongs to exactly one Project in v1 and changing membership requires an explicit auditable move, preserving an unambiguous workspace boundary.
- (decision) Every v1 Project has one canonical Project Folder; it may be created with the Project name when absent and may only be replaced explicitly, giving each workspace a stable folder anchor while Hexalith.Folders retains ownership.
- (decision) Project lifecycle is limited to Active and Archived in v1, with archived Projects retained for audit and excluded from automatic resolution by default, balancing history with safe context selection.
- (decision) Project creation requires only a user-supplied name and defaults to Active, but inferred creation always requires user confirmation, keeping onboarding light without allowing silent durable changes.
- (decision) Project Resolution returns NoMatch, SingleCandidate, or MultipleCandidates with reason codes and never silently attaches an ambiguous match, because correctness and explainability outweigh automatic attachment rate.
- (decision) Project Context contains authorized references, setup, and metadata rather than foreign payloads, and reports excluded or stale resources, so Chatbot gains continuity without hidden contamination.
- (decision) Project Setup governs conversation behavior and context policy rather than model-provider internals, and rejects secrets, unrestricted paths, and foreign payloads, keeping configuration durable, additive, and serialization-tolerant.
- (decision) Audit events, diagnostics, and operator inspection are metadata-only, authorization-gated, and tenant-scoped, enabling troubleshooting without exposing transcripts, files, prompts, secrets, or memory payloads.
- (decision) The MVP excludes content indexing, transcript or memory-payload storage, standalone end-user UI, generic project-management workflows, and cross-tenant sharing, keeping delivery focused on project context coordination.
- (decision) Success is measured by usable context availability, useful resolution or new-project proposals, zero unauthorized context leakage, and p95 under 500 ms for key metadata operations; the latency target is internal rather than an external SLA.
- (decision) Keep FR-22 scoped to operator read access; add FR-24 for bounded Safe Diagnostic Export; keep consequential action authority in the action-specific FRs and define a cross-cutting role/operation matrix, preserving stable requirement ownership.
- (decision) Adopt product roles Project User, Tenant Operator, Tenant Project Administrator, and delegated Service/Workflow Caller; restrict user-intent confirmations to Project Users through Chatbot, safe export to separately authorized operator/admin roles, relink/unlink administration to Project Users and Tenant Project Administrators, and prohibit surface choice or service identity from expanding authority.
- (decision) Use durable task states Pending, Running, WaitingForDependency, NeedsAttention, Succeeded, Rejected, Failed, and Cancelled; Succeeded/Rejected/Failed/Cancelled are terminal, NeedsAttention is nonterminal, Blocked is not a distinct state, timeouts/lost responses do not mutate state, and cancellation stops at the irreversible commit point.
- (decision) Adopt internal-launch performance envelope: 10,000 Projects per Tenant, 5,000 Context References per Project excluding its canonical Folder, 100,000 retained audit records per Project, cursor pages default 50/max 200, metadata-read p95 under 500 ms at 1,000 Projects and 500 references, p95 under 1 second at supported maximum, and durable-task admission p95 under 500 ms under authenticated warm steady-state with dependencies available.
- (decision) Adopt reliability envelope: 99.9% monthly authenticated API/task-admission availability excluding planned maintenance; committed-event RPO 0 within the primary-region durability domain after EventStore acknowledgement; regional DR governed by platform policy; 15-minute service RTO after process/node failure with dependencies healthy; durable tasks resume or reach truthful NeedsAttention within 5 minutes; trust-bearing mutations fail closed on dependency uncertainty; only idempotent retries are allowed and exhaustion cannot duplicate effects.
- (decision) Adopt retention/replay envelope: active tasks pollable until terminal; terminal results and scoped idempotency records retained 30 days or through associated result expiry; equivalent retries return the original task and changed payloads conflict; preview/confirmation artifacts expire after 15 minutes, are single-use, and invalidate on resource-version change; audit metadata retained at least 365 days subject to longer event-history obligations; resolution traces and safe exports are not persisted.
- (decision) Adopt security/key-management envelope: platform-approved authenticated encryption in transit and platform-managed encryption at rest; Projects owns no private keys and relies on approved KMS/secret-provider rotation/revocation evidence; customer-managed keys are out of MVP; authority is server-derived or cryptographically bound; trust-bearing mutations fail closed on stale/unknown/rebuilding/unavailable evidence; confirmations are tamper-evident and replay-protected; release evidence covers encryption, key availability, cross-tenant denial, expiry, tampering, and replay without secret exposure.
- (decision) Adopt overload/dependency envelope: per Tenant 100 metadata reads/s burst 200 and 20 mutation admissions/s burst 40; maximum 1,000 nonterminal tasks and 2 concurrent safe exports per Tenant; interactive dependency timeout 2 seconds and durable-step timeout 10 seconds; idempotent dependency calls retry at most 3 times within 30 seconds before WaitingForDependency/NeedsAttention; overload/unavailability returns structured retry guidance; accepted tasks are never dropped or duplicated.
- (decision) Adopt WCAG 2.2 AA for Chatbot candidate/confirmation/cancellation/recovery/task journeys and operator read/preview/mutation/export journeys; require keyboard operation, visible/restored focus, non-color status/error announcements, no preselected ambiguous candidate, usability at 200% zoom and 320 CSS-pixel width, and automated plus authenticated manual keyboard/screen-reader evidence.
- (decision) Adopt FR-24 Safe Diagnostic Export policy: separately authorized Tenant Operators/Admins may export through Web/CLI/MCP but not Chatbot; every attempt/outcome is metadata-only audited; global complete-export caps are 1 MiB encoded including envelope, 500 references, and 100 newest-first audit rows; deterministic ordering/truncation reports counts without excluded detail; no continuation cursor; upstream gaps are safe and explicit; generated exports are not retained.
- (decision) For folderless/invalid-Folder restore, require an authorized existing replacement or same-name Folder creation in the server preview before activation; bind Folder choice and Project version into confirmation; if Folder creation succeeds but activation cannot commit, use NeedsAttention plus reconciliation, never automatically delete Folders-owned resources, expose a safe recovery path, and audit metadata-only outcomes.
- (decision) Require server preview plus single-use confirmation for archive, restore, Conversation move/relink, Project Folder replacement, every unlink, ambiguous resolution confirmation, and inferred proposed-Project creation; explicitly actor-selected additive Conversation/File/Memory links use idempotent durable tasks without second confirmation, while inferred additive links require confirmation.
- (decision) Pending creation is visible only through durable-task surfaces, never Project list/open/resolution/context; initiators may poll and safely cancel their tasks; Operators/Admins may inspect metadata-only status and Admins may reconcile; reserved Project identity stays hidden until read-model-confirmed activation; NeedsAttention exposes safe reason codes/actions; task visibility follows the approved retention policy.
- (decision) Durable metadata-only audit covers consequential task admission/terminal outcome, confirmation use/cancel/stale/replay/tamper/auth denial, Project mutations and confirmed outcomes, manual reconciliation/NeedsAttention resolution, safe export, and stable upstream receipt IDs; intermediate task states, polls, retries, latency, SignalR, unused expiry, and read-only resolution traces are operational telemetry; equivalent idempotent retries do not duplicate audit.
- (decision) Adopt release thresholds: zero Active folderless Projects and zero unauthorized disclosures; all restart/duplicate/concurrency/lost-response acceptance tasks reach correct terminal or truthful NeedsAttention within recovery targets; create/restore require read-model confirmation; performance/accessibility/security/deployment/smoke/rollback/stakeholder evidence passes; failed critical tests or unexplained critical skips block release and unavailable environments remain not verified.
