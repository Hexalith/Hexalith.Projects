## Industry Analysis

### Domain Context and Fit

Hexalith.Projects is not a generic project management database. The PRD defines it as a tenant-aware AI workspace boundary that helps Chatbot determine project membership, retrieve project context, and propose project creation when no suitable project exists. The module owns project identity, metadata, setup, lifecycle, references to other bounded contexts, resolution outcomes, and metadata-only audit evidence. It explicitly does not own conversation transcripts, file contents, raw prompts, secrets, memory payloads, or unrestricted filesystem paths.

That domain shape fits event sourcing well because the business questions are temporal and diagnostic: who created or changed a Project, which references were linked or removed, why a candidate was selected or rejected, whether an action was confirmed, and what metadata-only audit evidence exists. Microsoft describes CQRS as separating write and read models; it also notes that an event store can act as the source of truth and that materialized views can be regenerated by replaying historical events. That directly matches Projects' need for authoritative state changes plus multiple operational read surfaces. Source: https://learn.microsoft.com/en-us/azure/architecture/patterns/cqrs

### EventStore Persistence Pattern

The relevant persistence pattern is stream-per-aggregate, not table-per-view. Kurrent/EventStoreDB documentation describes EventStoreDB as purpose-built for event storage, where state changes are stored as independent events, logically organized into streams, typically one stream per entity. It also documents expected-revision checks for append concurrency: the client supplies the stream state/version it expects, and append fails when the stream is not in that state. Source: https://docs.kurrent.io/server/v23.10/features/streams and https://docs.kurrent.io/clients/java/legacy/v5.4/appending-events

Hexalith.EventStore already exposes the equivalent internal shape. `AggregateIdentity` is the canonical `{tenant}:{domain}:{aggregateId}` tuple and derives actor IDs, event stream key prefixes, metadata keys, snapshot keys, pub/sub topics, and queue session IDs. `EventMetadata` carries structured envelope metadata such as message ID, aggregate ID/type, tenant, domain, sequence number, global position, timestamp, correlation ID, causation ID, user ID, domain service version, event type, metadata version, and serialization format. `DomainResult` carries only event payloads and rejects mixed success/rejection results.

Implication for Projects: project state should be persisted by submitting commands through the Hexalith.EventStore command pipeline to a Projects aggregate, which emits payload events. Hexalith.EventStore should assign envelope metadata, sequence numbers, positions, correlation/causation data, and publish only after persistence. Domain code should not write directly to Dapr state, Redis, PostgreSQL, Cosmos DB, brokers, or EventStore metadata.

### Recommended Write Model

The default aggregate should be one Project aggregate per Project:

- Domain: `projects`
- Aggregate ID: stable `ProjectId`
- EventStore identity: `{tenant}:projects:{projectId}`
- Actor ID/state keys/topics: derived only from `AggregateIdentity`

This aggregate should own project-local invariants:

- A Project is created once and starts active unless rejected by validation.
- Project name/setup updates are explicit and metadata-only.
- Archival prevents normal context retrieval/linking but preserves auditability.
- v1 has exactly one Project Folder, replaceable only through explicit action.
- Conversation membership is single-project in v1; moving requires an explicit move command.
- File and Memory references are links only, never payload copies.
- Ambiguous resolution confirmation is explicit; inferred proposals do not create Projects until confirmed.

Candidate commands:

- `CreateProject`
- `UpdateProjectSetup`
- `ArchiveProject`
- `LinkConversation`
- `MoveConversation`
- `SetProjectFolder`
- `LinkFileReference`
- `UnlinkFileReference`
- `LinkMemory`
- `UnlinkMemory`
- `ConfirmProjectResolution`

Candidate events:

- `ProjectCreated`
- `ProjectSetupUpdated`
- `ProjectArchived`
- `ConversationLinked`
- `ConversationMoved`
- `ProjectFolderSet`
- `FileReferenceLinked`
- `FileReferenceUnlinked`
- `MemoryLinked`
- `MemoryUnlinked`
- `ProjectResolutionConfirmed`
- Rejection events such as `ProjectCreationRejected`, `ProjectReferenceLinkRejected`, `ProjectArchivedCommandRejected`, `ProjectResolutionConfirmationRejected`

These names follow local rules: command records use imperative names without a `Command` suffix; event records use past-tense names without an `Event` suffix; rejection events implement `IRejectionEvent`.

### Metadata and Envelope Strategy

CloudEvents is useful as a reference model for why envelope metadata should be standardized and kept separate from event payloads. The CloudEvents specification requires fields such as `id`, `source`, `specversion`, and `type`, and states that producers must ensure `source` + `id` is unique for distinct events. Source: https://github.com/cloudevents/spec/blob/main/cloudevents/spec.md

Hexalith.EventStore has its own envelope contract, so Projects should not adopt CloudEvents directly inside domain payloads. The takeaway is architectural: keep routing, identity, tracing, causation, serialization, and producer metadata in the envelope; keep Projects business facts in event payloads.

Projects event payloads should contain safe business metadata only:

- Project name, description, lifecycle state, and setup policy fields.
- Stable external reference IDs and bounded-context names.
- Authorization/freshness evidence fingerprints or reason codes, not tokens or raw evidence bodies.
- Resolution reason codes, confidence bands, candidate IDs, and explicit confirmation metadata.
- Actor/source identifiers only when allowed by the existing envelope or safe audit model.

Projects event payloads should not contain:

- Conversation transcript content.
- File contents, unrestricted filesystem paths, provider secrets, or raw provider metadata.
- Memory payloads.
- Raw prompts or model-provider internals.
- Event envelope fields owned by Hexalith.EventStore.

### Projection and Query Model

Projects needs multiple read shapes, so projections should be derived from the authoritative event streams rather than written as primary state. EventStoreDB documentation describes subscriptions for reacting to appended events and system projections that link events by category or event type. Source: https://docs.kurrent.io/server/v22.10/persistent-subscriptions.html and https://docs.kurrent.io/server/v24.10/features/projections/system.html

Inside Hexalith, the project context rules are more specific: use `CachingProjectionActor`, ETag actors, projection notifiers, and the existing EventStore query/projection infrastructure before inventing custom read-side routing. That means Projects should define projection contracts for:

- Project list by tenant, lifecycle, warning state, and update timestamp.
- Project detail by tenant/project ID.
- Reference inventory by project and by external reference ID.
- Resolution diagnostics by resolution case/correlation ID.
- Audit timeline by project and tenant.
- Conversation-start setup view, filtered for authorization and availability.

Projections should carry watermarks/ETags/freshness metadata so Query handlers can distinguish fresh, stale, unavailable, and fail-closed states. This aligns with the UX requirement for consistent states and reason codes across CLI, MCP, and Web.

### Dapr Runtime Boundary

Dapr's current actor docs remain relevant because Hexalith.EventStore uses Dapr actors/state/pub-sub as the infrastructure abstraction. Dapr documents actors as a building block with actor state management, and the state API states that actor state stores require transactional support and strong consistency when backed by a distributed database. Source: https://docs.dapr.io/developing-applications/building-blocks/actors/ and https://docs.dapr.io/reference/api/state_api/

Implication for Projects: Dapr is a runtime boundary owned by EventStore/hosting infrastructure, not by Projects domain logic. Projects aggregate code should be pure `Handle(...)` and `Apply(...)` logic. Hosts can register aggregates/projections and wire Dapr/EventStore services, but project contracts and domain packages should not take direct Dapr or database dependencies.

### Market/Technology Dynamics Relevant to the Decision

The "market" issue here is not vendor selection; Hexalith already standardizes on Hexalith.EventStore. The relevant technology dynamics are the common costs and benefits of event-sourced CQRS systems:

- Benefit: authoritative audit trail, temporal reconstruction, clear command history, and projection rebuildability.
- Benefit: multiple read models can serve CLI, MCP, Web, Chatbot, and operator diagnostics without weakening the write model.
- Cost: schema evolution, projection rebuilds, privacy controls, and eventual-consistency/freshness semantics require deliberate design.
- Risk: treating read models as write authority creates drift and cross-tenant leakage risk.

Academic research on event-sourced systems identifies recurring industry challenges: event-system evolution, steep learning curve, technology availability, rebuilding projections, and data privacy. Source: https://arxiv.org/abs/2104.01146

For Hexalith.Projects, those challenges translate into concrete guardrails:

- Keep events additive and backward-compatible; do not create `V2` events casually.
- Maintain deserialization compatibility for every event type produced.
- Add projection rebuild tests and operational evidence for project list/detail/reference/audit projections.
- Keep sensitive payloads out of Projects events so privacy operations remain tractable.
- Make freshness explicit in query responses instead of silently returning stale context.

### Initial Recommendation

Use Hexalith.EventStore as the sole write authority for Projects. Model Project as the primary aggregate, persist only metadata-safe Project events, and build tenant-scoped projections for every query surface. Commands should validate against rehydrated aggregate state and safe authorization/freshness evidence; successful commands emit Project events, rejected commands emit rejection events, and no-op idempotent commands return no events. CLI, MCP, Web, and Chatbot should read through EventStore query/projection contracts rather than bypassing the command/query pipeline.

The strongest immediate design decision is to keep `ProjectContext` as an assembled read result, not a separately mutable aggregate. The Project aggregate owns references and setup metadata; projections/query handlers assemble context by combining Project state with authorization-filtered metadata from Conversations, Folders, and Memories. This preserves bounded-context ownership while still giving Chatbot a coherent Project Context.

## Competitive Landscape

### Key Players and Ecosystem Roles

For Hexalith.Projects, the useful competitive landscape is an architecture ecosystem map:

- **Hexalith.EventStore** is the selected platform foundation and should remain the write authority.
- **EventStoreDB/Kurrent** is the closest external reference model for event-store semantics: event streams, stream metadata, subscriptions, projections, expected-revision appends, and append-only event history.
- **Marten** is a relevant .NET alternative pattern because it combines PostgreSQL document storage with event sourcing and projection infrastructure.
- **Axon Framework/Axon Server** is a mature Java ecosystem reference for event-sourced aggregates, repositories, command handling, and event stores.
- **Kafka and streaming platforms** are relevant as messaging/log-processing competitors, but they should not become the first-point-of-contact aggregate event store for Projects.
- **Dapr** is not a competing persistence model inside Hexalith; it is the runtime abstraction Hexalith uses for actors, state, pub/sub, and sidecar-managed infrastructure.

EventStoreDB's position is strongest where immutable streams, stream reads, optimistic concurrency, and subscriptions are the core requirement. Marten's position is strongest where a .NET team wants PostgreSQL-backed event storage plus document projections in one database. Axon's position is strongest in JVM systems that want an integrated DDD/CQRS/event-sourcing framework. Kafka's position is strongest for high-throughput event streaming and integration pipelines, not aggregate invariant enforcement.

Sources: https://docs.kurrent.io/server/v23.10/features/streams, https://martendb.io/events/projections/, https://docs.axoniq.io/axon-framework-reference/main/events/infrastructure/, https://docs.dapr.io/developing-applications/building-blocks/pubsub/pubsub-overview/

### Market Share and Competitive Positioning

No reliable, directly comparable market-share data was found for "event stores" as a narrow category. The better decision axis is capability fit:

- **Purpose-built event store:** EventStoreDB/Kurrent emphasizes streams, append semantics, metadata, projections, and subscriptions. This is the closest external benchmark for Hexalith.EventStore.
- **Database-backed event sourcing:** Marten positions PostgreSQL as both event store and document database, with projections converting event streams into queryable documents or flat tables.
- **Framework-integrated CQRS/event sourcing:** Axon integrates aggregate modeling, event store access, command handling, event processors, and messaging concepts in the JVM ecosystem.
- **Streaming/log infrastructure:** Kafka is widely used for event streaming, data integration, and stream processing, but it is a different category from an aggregate event store.
- **Runtime abstraction:** Dapr offers platform-agnostic APIs and at-least-once pub/sub delivery, making it useful for delivery and hosting concerns after the event-store write is committed.

This reinforces Hexalith's existing direction: use Hexalith.EventStore for command-side persistence and use Dapr-provided delivery as infrastructure, not as the source of truth.

Sources: https://martendb.io/events/projections/aggregate-projections.html, https://docs.axoniq.io/axon-framework-reference/4.13/axon-framework-commands/modeling/aggregate/, https://docs.dapr.io/reference/api/pubsub_api/

### Competitive Strategies and Differentiation

The major patterns differentiate on where they put consistency, projection ownership, and infrastructure coupling:

- **EventStoreDB-style stream store:** Strong fit for aggregate streams, expected-revision concurrency, append-only history, stream reads, and subscriptions. Differentiation is event-store semantics rather than general query flexibility.
- **Marten-style PostgreSQL event store:** Strong fit for .NET/PostgreSQL shops that want event streams and projections in one operational database. Differentiation is tight event/document integration.
- **Axon-style framework:** Strong fit for teams that want a full application framework around aggregates, command handlers, event sourcing handlers, event processors, and event storage. Differentiation is integrated programming model.
- **Kafka-style event streaming:** Strong fit for high-throughput distribution, fan-out, stream processing, and integration. Differentiation is scalable log/messaging infrastructure, not aggregate replay with per-stream concurrency checks.
- **Dapr-style abstraction:** Strong fit for portable actors, state, service invocation, pub/sub, resiliency, and sidecar-managed infrastructure. Differentiation is operational portability across backing components.

Hexalith.EventStore's differentiation inside Hexalith is that it combines the needed event-store semantics with the local tenant model, envelope metadata, command pipeline, projection actors, admin API, CLI, MCP, testing utilities, and Dapr runtime conventions already used by sibling modules.

Sources: https://docs.kurrent.io/clients/java/legacy/v5.4/appending-events, https://martendb.io/events/projections/, https://docs.axoniq.io/axon-framework-reference/main/events/infrastructure/

### Business Models and Value Propositions

For Hexalith.Projects, the value proposition is internal platform leverage rather than external procurement:

- **Hexalith.EventStore value:** one consistent persistence and projection model across Hexalith modules; tenant identity and envelope rules are already established.
- **EventStoreDB/Kurrent value:** validates that stream-per-entity, metadata separation, expected revision, and subscription patterns are mainstream event-store mechanics.
- **Marten value:** validates rich projection/read-model strategies, especially aggregate and flat projections, but it would introduce direct PostgreSQL coupling that conflicts with Hexalith's current Dapr/EventStore boundary for domain services.
- **Axon value:** validates aggregate reconstruction and event-sourced command handling, but it is JVM-framework centric and not directly aligned with Hexalith's .NET package architecture.
- **Kafka value:** validates event distribution and stream-processing use cases, but should remain downstream/transport infrastructure if introduced, not authoritative Projects persistence.
- **Dapr value:** gives Hexalith portability and consistent runtime APIs, with at-least-once delivery semantics that require idempotent consumers and projection handlers.

The practical conclusion is to avoid adding Marten, Axon, Kafka, or direct database dependencies to the Projects write model. Their patterns can inform design, but Hexalith.EventStore is the implementation spine.

Sources: https://docs.dapr.io/developing-applications/building-blocks/pubsub/pubsub-overview/, https://martendb.io/events/projections/, https://docs.axoniq.io/axon-framework-reference/4.13/axon-framework-commands/modeling/aggregate/

### Competitive Dynamics and Entry Barriers

The main entry barriers for a Projects persistence implementation are not licensing or vendor access. They are architectural consistency:

- **Tenant isolation:** every key, actor, stream, topic, projection, query, and log field must derive from the canonical EventStore identity model.
- **Envelope ownership:** Projects must not populate or spoof EventStore metadata fields.
- **Payload safety:** Projects must not persist transcripts, file contents, raw prompts, secrets, memory payloads, or unsafe summaries.
- **Projection correctness:** read-side views must be rebuildable and must expose freshness/watermark state.
- **Idempotency and duplicate delivery:** Dapr pub/sub and event delivery are at-least-once patterns, so consumers and projection handlers must tolerate duplicate or retried messages.
- **Schema evolution:** event payloads must remain additive and backward-compatible.
- **Operational parity:** CLI, MCP, and Web must expose consistent reason codes, lifecycle states, warnings, and audit identifiers.

These barriers argue against a quick CRUD persistence layer. A table-first implementation would be cheaper initially but would fight the audit, replay, projection, and tenant-isolation requirements that Projects already has.

Sources: https://docs.dapr.io/reference/api/pubsub_api/, https://arxiv.org/abs/2104.01146

### Ecosystem and Partnership Analysis

The Hexalith ecosystem boundaries should be explicit:

- **Projects -> EventStore:** state-changing operations go through EventStore command submission. Projects aggregates emit payload events only.
- **Projects -> Tenants:** tenant context and authorization evidence are consumed as safe metadata; Projects does not become the tenant authority.
- **Projects -> Conversations:** store stable conversation references and metadata only; do not store transcripts.
- **Projects -> Folders:** store Project Folder and File Reference identities/metadata only; do not store file contents or bypass folder authorization.
- **Projects -> Memories:** store memory references and metadata only; do not store memory payloads.
- **Projects -> Chatbot:** expose project context and resolution outputs as metadata-rich read results; Chatbot owns conversation experience and model orchestration.
- **Projects -> FrontComposer:** Web UI should be generated/composed through the established FrontComposer model.
- **Projects -> CLI/MCP:** access should go through Admin/API/query contracts, not Dapr-direct calls.

This ecosystem map makes the persistence boundary clear: Projects persists its own metadata and associations as events; it never imports another bounded context's payload data into its event stream.

### Competitive Implications for Implementation

The most defensible implementation strategy is:

1. Build Projects on Hexalith.EventStore aggregate/projection packages, not on a new database abstraction.
2. Use external ecosystems as pattern validation only: EventStoreDB for stream/concurrency semantics, Marten for projection design ideas, Axon for aggregate reconstruction discipline, Kafka/Dapr for delivery semantics.
3. Keep Projects write paths narrow and event-sourced; put operational query needs into projections.
4. Use Dapr pub/sub and projection notifications after persistence succeeds; never publish first.
5. Treat direct SQL/document/streaming dependencies in Projects domain packages as architectural regressions.

Confidence: high for the recommendation to use Hexalith.EventStore as the write authority, because it is supported by local architecture rules and current event-sourcing sources. Confidence: medium for exact projection names and event catalogs, because those should be validated in architecture/story design.

## Regulatory Requirements

Hexalith.Projects persistence is primarily affected by privacy, cybersecurity, AI governance, and auditability requirements rather than industry-specific licensing. The key regulatory posture is: store the minimum metadata needed to support Project Context, resolution, references, lifecycle, and audit; keep payload-bearing data in the owning bounded contexts.

### Compliance Frameworks

The compliance framework for Projects persistence should be a layered control model:

- **Privacy controls:** metadata-only event payloads, explicit purpose for every persisted field, no cross-context payload import, and retention strategy for Project events/read models.
- **Security controls:** tenant-scoped aggregate identity, authorization-gated command/query APIs, no direct Dapr access from CLI/MCP clients, and no direct database dependencies in domain packages.
- **Audit controls:** immutable Project events, command correlation/causation metadata owned by EventStore, metadata-only audit timelines, and queryable operation evidence.
- **Operational controls:** projection watermarks, ETags, rebuild checkpoints, replay procedures, dead-letter handling, and operator-safe diagnostics.
- **Governance controls:** documented event catalog, projection catalog, allowed payload fields, redaction policy, schema evolution policy, and test evidence.

Hexalith.EventStore already supplies several structural controls: canonical aggregate identity, event envelope metadata, command pipeline, projection actors, ETag actors, notifiers, admin API, and testing utilities. Projects should use these controls directly instead of creating a parallel compliance mechanism.

Sources: https://www.nist.gov/cyberframework/index.cfm, https://www.iso.org/standard/27001

### Data Protection and Privacy

The core privacy requirement is data minimization. The European Commission explains GDPR data minimization as collecting personal data that is adequate, relevant, and limited to what is necessary for the purpose. The FTC similarly advises businesses to collect only what they need, keep it safe, and dispose of it securely. Sources: https://commission.europa.eu/law/law-topic/data-protection/rules-business-and-organisations/principles-gdpr/how-much-data-can-be-collected_en and https://www.ftc.gov/business-guidance/privacy-security

For Projects, the minimum viable persistent data set should be:

- Project identity, tenant, lifecycle state, display metadata, and setup metadata needed for Chatbot continuity.
- Stable reference IDs for Conversations, one Project Folder, File References, and Memories.
- Safe reference metadata: owning bounded context, resource type, lifecycle/availability status, authorization/freshness status, and reason codes.
- Resolution evidence: candidate IDs, reason codes, confidence band, ambiguity status, confirmation status, and correlation IDs.
- Metadata-only audit evidence: operation type, affected reference IDs, actor/source identifier where allowed, timestamp, correlation ID, and result.

Data that should remain outside Projects:

- Full conversation transcripts and summaries that reveal transcript content.
- File contents, unrestricted paths, provider secrets, raw Git metadata, and credentials.
- Memory payloads.
- Raw prompts, model responses, system instructions not explicitly part of safe Project Setup, and model-provider internals.
- Raw JWTs, tokens, authorization artifacts, and unredacted command bodies.

Event sourcing creates a retention tension: immutable events are excellent for audit and replay, but harder to erase selectively. Projects reduces this risk by avoiding personal/payload data in events. If future stories require personal data in Project events, they should trigger a privacy design review, retention rule, protection metadata strategy, and deletion/crypto-shredding approach before implementation.

### Implementation Considerations

Projects persistence should adopt these compliance-oriented engineering decisions:

- **Event catalog gate:** every new event must list purpose, fields, sensitivity classification, owner, retention expectation, and allowed consumers.
- **No payload import gate:** tests must fail if Project events include transcript snippets, file contents, memory payloads, raw prompts, secrets, unrestricted paths, or raw tokens.
- **Tenant isolation gate:** tests must verify tenant/domain/aggregate key derivation and query-side authorization filtering.
- **Envelope separation gate:** domain services return event payloads only; EventStore owns metadata.
- **Projection freshness gate:** query responses must expose stale/unavailable/fail-closed states rather than hiding projection lag.
- **Idempotency gate:** command processing and projection handlers must tolerate retries and at-least-once delivery.
- **Audit evidence gate:** every mutating command should produce either a domain event, rejection event, or explicit no-op with command status evidence.
- **Schema evolution gate:** events are additive and tolerant; no `V2` event type unless approved by architecture.
- **Logging gate:** logs use IDs, reason codes, tenant, correlation, and status; never payloads or command bodies.

These gates should become acceptance criteria in implementation stories and should be backed by Tier 1 tests where possible.

Sources: https://owasp.org/www-project-application-security-verification-standard/, https://docs.dapr.io/developing-applications/building-blocks/pubsub/pubsub-overview/

### Risk Assessment

High risks:

- **Payload leakage into immutable events:** strongest privacy and compliance risk. Mitigation: metadata-only event schema, negative tests, code review checklist, and redaction-safe logging.
- **Cross-tenant projection leakage:** high security risk because Projects is a workspace boundary. Mitigation: canonical identity derivation, tenant-scoped projection keys, query-side authorization tests, and fail-closed defaults.
- **Read-model treated as authority:** high consistency risk. Mitigation: state changes only through EventStore commands; projections are derived and rebuildable.
- **AI governance evidence gap:** medium-to-high risk if Chatbot use cases fall under AI transparency or high-risk obligations. Mitigation: persist safe resolution/confirmation metadata and correlation IDs, but keep prompts/outputs outside Projects.

Medium risks:

- **Projection staleness causing unsafe context assembly:** mitigation: freshness metadata, ETags, watermarks, and fail-closed query responses.
- **Event schema evolution breaking replay:** mitigation: additive contracts, compatibility tests, and replay tests for old event samples.
- **Over-retention in read models:** mitigation: projection retention policy and rebuild capability from authoritative events.

Low-to-medium risks:

- **Certification evidence gaps:** mitigation: preserve implementation artifacts and test reports as audit evidence.
- **Ambiguous ownership between Projects and sibling modules:** mitigation: bounded-context reference map and explicit non-ownership rules in event catalog.

Overall regulatory posture: Projects can be compliance-friendly if it remains metadata-only, event-sourced, tenant-scoped, and projection-driven. The biggest mistake would be storing convenience copies of payload-bearing data inside Projects events.

## Technical Trends and Innovation

### Emerging Technologies

The strongest technical trend for Projects persistence is not a new database. It is the convergence of event-sourced persistence, standardized telemetry, portable runtime infrastructure, and AI-governance evidence.

Relevant emerging or maturing technologies:

- **Event-store projections and subscriptions:** EventStoreDB/Kurrent continues to emphasize system projections, persistent subscriptions, catch-up subscriptions, and linked streams by event type/category/correlation. These patterns support rebuilding and operating derived read models. Sources: https://docs.kurrent.io/server/v24.10/features/projections/system.html and https://docs.kurrent.io/server/v22.10/persistent-subscriptions.html
- **OpenTelemetry semantic conventions:** OpenTelemetry defines common names for traces, metrics, logs, profiles, and resources. This matters because Projects resolution, command processing, projection updates, and context assembly need consistent observability attributes across services. Source: https://opentelemetry.io/docs/concepts/semantic-conventions/
- **Dapr resiliency and runtime building blocks:** Dapr supports resiliency policies such as retries, timeouts, and circuit breakers across apps, components, and actors. It also positions actors, pub/sub, state management, workflow, secret stores, jobs, distributed lock, and cryptography as portable building blocks. Sources: https://docs.dapr.io/operations/resiliency/resiliency-overview/ and https://dapr.io/
- **CloudEvents-style event metadata standardization:** CloudEvents has graduated in CNCF and standardizes event metadata for cross-platform interoperability. - **AI tool protocol standardization:** MCP formalizes structured tool/resource interaction with JSON-RPC semantics. Projects' MCP surface should therefore expose structured metadata and errors, not prose-only diagnostics or domain payloads. Source: https://modelcontextprotocol.io/specification/2025-06-18/basic/index

Implication: Projects should invest in stable metadata schemas, correlation IDs, reason codes, projection watermarks, and structured tool/query responses. These are now core platform capabilities, not optional observability polish.

### Digital Transformation

For Hexalith.Projects, digital transformation means turning project continuity into a durable operational control plane. The UX research already requires CLI, MCP, and FrontComposer Web surfaces over a shared diagnostic model. Current technology trends support that:

- Event sourcing provides historical state reconstruction and auditability.
- CQRS projections provide task-specific read models for list/detail/reference/resolution/audit views.
- OpenTelemetry provides cross-service trace correlation for command and query paths.
- Dapr provides platform-agnostic runtime APIs and resiliency policies.
- MCP provides structured agent-facing diagnostics and maintenance operations.

The transformation opportunity is to make Project state explainable. Instead of only returning "context found" or "context failed," Projects can return safe evidence: selected project, candidate set, reason codes, excluded references, authorization/freshness state, projection watermark, and audit event ID.

This aligns with AI governance trends. The EU AI Act includes record-keeping/logging concepts for high-risk AI systems, and the official AI Act Explorer exposes provisions around risk, transparency, and logging. Projects should not try to be the AI compliance system for Chatbot, but it can provide metadata evidence that supports governance workflows. Sources: https://ai-act-service-desk.ec.europa.eu/en/ai-act-explorer and https://eur-lex.europa.eu/eli/reg/2024/1689/oj

### Innovation Patterns

The relevant innovation patterns are practical and implementation-level:

- **Metadata-first event design:** Persist stable identifiers, reason codes, fingerprints, timestamps, and statuses; avoid payload copies.
- **Projection-as-product:** Treat each read surface as an explicit projection contract with freshness, ETag, paging, filtering, and rebuild behavior.
- **Evidence-ready operations:** Mutating commands should produce a traceable chain: command status, event sequence, projection update, notification, and audit output.
- **Fail-closed context assembly:** If authorization, lifecycle, or freshness cannot be verified, return excluded/unavailable/stale metadata instead of silently including references.
- **Structured MCP diagnostics:** MCP resources/tools should return typed metadata and error codes, mirroring CLI/Web semantics.
- **Telemetry-aligned domain metadata:** Use consistent attribute names for tenant, project, aggregate, command, correlation, causation, projection, reason code, and freshness state.

This pattern set is more valuable than adopting a new product. It makes Hexalith.EventStore persistence observable, auditable, and safely consumable by Chatbot and operators.

Sources: https://opentelemetry.io/docs/specs/semconv/, https://modelcontextprotocol.io/specification/2025-06-18/basic/index

### Future Outlook

The next few years of event-driven and AI-assisted systems point toward more, not less, demand for:

- replayable system history,
- structured decision evidence,
- tenant-scoped access boundaries,
- cross-surface diagnostic parity,
- standardized telemetry,
- and governance-aware AI tool interactions.

Event sourcing remains valuable when the domain needs audit, reconstruction, and multiple derived views. It remains costly when teams event-source low-value CRUD data or store sensitive payloads in immutable streams. Projects is on the favorable side of that tradeoff because project context, reference changes, resolution decisions, and audit timelines are naturally temporal.

Projection technology is also moving toward more explicit operational controls: rebuild checkpoints, subscriptions, backpressure, and freshness indicators. Kurrent notes that some problems are better served by external read models populated by catch-up subscriptions rather than server-side projections alone. Source: https://docs.kurrent.io/server/v22.10/projections

Future implication: Projects should not assume one projection mechanism fits all read needs. Use Hexalith.EventStore `CachingProjectionActor` and query infrastructure where it fits; use explicit projection stores or workers where read models need broader indexing, backpressure control, or rebuild isolation.

### Implementation Opportunities

Priority opportunities for Projects:

1. **Define a Projects event catalog before code.** Include event name, command source, fields, sensitivity classification, replay behavior, and projection consumers.
2. **Create a single `ProjectAggregate`.** Start with Project lifecycle, setup, Project Folder, Conversation link/move, File Reference link/unlink, Memory link/unlink, and resolution confirmation.
3. **Define read contracts as projections.** Minimum set: `ProjectListProjection`, `ProjectDetailProjection`, `ProjectReferenceIndexProjection`, `ProjectResolutionTraceProjection`, `ProjectAuditTimelineProjection`, and `ConversationStartSetupProjection`.
4. **Add projection freshness metadata.** Every query response should include watermark/ETag/freshness status where user-facing decisions depend on current state.
5. **Use OpenTelemetry-compatible semantic naming.** Standardize attributes for `tenant.id`, `project.id`, `eventstore.domain`, `eventstore.aggregate_id`, `command.type`, `projection.type`, `resolution.reason_code`, and `context.reference_state` or local equivalents.
6. **Make MCP and CLI schema-first.** Generate or share DTOs so MCP, CLI, Web, and Chatbot see the same reason codes and states.
7. **Build replay and leakage tests early.** Include aggregate replay, projection rebuild, metadata-only negative tests, and cross-tenant query tests in the first implementation slice.

### Challenges and Risks

Technical risks:

- **Over-modeling events:** too many narrow event types can increase evolution cost. Mitigation: model business facts, not UI actions.
- **Under-modeling references:** too generic a `ReferenceChanged` event can weaken invariants and projections. Mitigation: use distinct reference events for Conversation, Folder, File, and Memory when rules differ.
- **Projection sprawl:** every screen/tool wanting its own read model can create maintenance overhead. Mitigation: define a projection catalog and require ownership/rebuild tests.
- **Telemetry cardinality:** project IDs and reference IDs are useful for diagnostics but can create high-cardinality metric problems. Mitigation: use IDs in traces/logs; be selective in metrics.
- **MCP overreach:** agent tools may be tempted to perform multi-step orchestration. Mitigation: MCP maps intent to safe commands/queries; domain logic stays in aggregates and query handlers.
- **AI compliance drift:** AI risk classification can change as Chatbot behavior changes. Mitigation: Projects persists safe governance metadata but does not claim compliance for model decisions.
- **Dapr at-least-once delivery duplicates:** projections and subscribers must be idempotent. Mitigation: use event identity, sequence numbers, and projection watermarks.

Sources: https://docs.dapr.io/developing-applications/building-blocks/pubsub/pubsub-overview/, https://opentelemetry.io/docs/specs/semconv/general/

## Recommendations

### Technology Adoption Strategy

Use Hexalith.EventStore as the persistence spine and adopt surrounding trends only where they strengthen the existing architecture:

- Adopt OpenTelemetry semantic conventions for trace/log attribute consistency.
- Adopt Dapr resiliency policies at host/infrastructure boundaries.
- Adopt MCP schema discipline for structured agent-facing diagnostics.

### Risk Mitigation

Most important mitigations:

- Add a `NoPayloadLeakage` test suite that scans Project events, logs, query DTOs, and audit outputs for forbidden fields.
- Require tenant-isolation negative tests on every command and query path.
- Require replay tests for every emitted event type before the first release.
- Add idempotency tests for duplicate commands and duplicate projection deliveries.
- Define projection freshness semantics before Chatbot consumes Project Context.
- Keep a sample event stream and projection rebuild fixture under tests so event evolution failures are visible.

Technical conclusion: the winning pattern is conservative, not exotic. Projects should use event sourcing because the domain is temporal and audit-heavy, then compensate for event-sourcing complexity with strict metadata minimization, explicit projections, standardized telemetry, and schema-first operational surfaces.

## Research Synthesis: EventStore Persistence for Hexalith.Projects Module Data

### Executive Summary

Hexalith.Projects should implement persistence through Hexalith.EventStore, not through a new CRUD database, document store, or streaming log. The module's durable data is temporal by nature: Project creation, setup changes, reference links, context-resolution confirmations, archive decisions, and audit evidence all benefit from append-only history, replay, and projection rebuilds. Microsoft’s current event-sourcing guidance states that the event stream remains the source of truth and materialized views can be regenerated from it; this matches the Projects requirement for operational read models without weakening write authority. Source: https://learn.microsoft.com/en-us/azure/architecture/patterns/event-sourcing

The core design decision is: **one `ProjectAggregate` per Project, persisted as metadata-only events under `{tenant}:projects:{projectId}`; `ProjectContext` is an assembled read result, not a mutable aggregate.** Projects owns metadata and references, while Conversations, Folders, Memories, Chatbot, and Tenants retain ownership of their payloads and authority boundaries.

The most important implementation constraint is privacy and tenant isolation. Projects events must never contain conversation transcripts, file contents, memory payloads, raw prompts, secrets, unrestricted paths, raw tokens, or EventStore envelope fields. Every query surface should be projection-backed, tenant-scoped, authorization-filtered, and explicit about freshness, warnings, and reason codes.

**Key Findings:**

- Hexalith.EventStore already provides the canonical identity, event envelope, command pipeline, Dapr actor/state/pub-sub integration, projection/query infrastructure, admin API, CLI/MCP support, and testing utilities needed by Projects.
- External ecosystems validate the pattern: EventStoreDB/Kurrent for stream/concurrency/subscription semantics, Marten for projection design, Axon for aggregate reconstruction discipline, Dapr for runtime abstractions, OpenTelemetry for telemetry standardization, and MCP for structured tool contracts.
- Compliance pressure reinforces metadata minimization, immutable audit evidence, purpose limitation, projection freshness, and explicit tenant boundaries.
- The highest-risk failure mode is storing convenience copies of payload-bearing data in immutable Project events.

**Strategic Recommendations:**

- Ratify `ProjectAggregate` as the sole write authority for Project lifecycle, setup, references, and resolution confirmation.
- Define a Projects event catalog before implementation, including field classification and forbidden payload rules.
- Build projection contracts for list, detail, references, resolution trace, audit timeline, and conversation-start setup.
- Make freshness/watermark metadata part of every query contract that can influence Chatbot context.
- Add tenant-isolation, replay, idempotency, metadata-leakage, and projection-rebuild tests in the first slice.

### 2. Domain Overview and EventStore Fit

Projects is a metadata control plane for AI workspace continuity. It helps Chatbot identify project membership, retrieve context references, resolve ambiguous context, and preserve continuity across conversations, folders, files, and memories. It is not a generic project-management system and should not store task boards, transcripts, file contents, memory payloads, raw prompts, or model-provider internals.

This domain is a strong event-sourcing fit because the most important facts are changes over time:

- Project was created.
- Setup was updated.
- Conversation was linked or moved.
- Project Folder was set.
- File or Memory reference was linked or unlinked.
- Project was archived.
- Resolution was ambiguous, proposed, or confirmed.
- Audit evidence was produced.

The event stream should answer "what happened and why"; projections should answer "what should this user/tool see now."

### 3. Recommended Persistence Model

Recommended aggregate:

- Aggregate: `ProjectAggregate`
- Domain: `projects`
- Aggregate ID: `ProjectId`
- EventStore identity: `{tenant}:projects:{projectId}`
- Source of truth: Project event stream
- Read authority: derived projections only

State owned by the aggregate:

- Project identity and lifecycle.
- Project display/setup metadata.
- Single v1 Project Folder reference.
- Conversation membership reference and move history.
- Optional File Reference links.
- Optional Memory links.
- Resolution confirmation facts.
- Idempotency fingerprints or equivalent dedupe state where needed.

State not owned by the aggregate:

- Tenant authority and tenant lifecycle.
- Conversation transcript or conversation payload.
- Folder/file contents or provider credentials.
- Memory content.
- Chatbot prompt/model orchestration.
- EventStore envelope metadata.

The aggregate should expose pure `Handle(...)` methods and replay state through pure `Apply(...)` methods, using Hexalith.EventStore conventions. Infrastructure belongs in host/server layers, not in contracts/domain packages.

### 5. Projection and Query Model

Recommended projection set:

- `ProjectListProjection`: tenant-scoped list for Active/Archived Projects, warnings, updated timestamp, and filters.
- `ProjectDetailProjection`: lifecycle, setup metadata, safe identifiers, reference summary, warnings.
- `ProjectReferenceIndexProjection`: lookup by conversation/folder/file/memory reference.
- `ProjectResolutionTraceProjection`: candidate Projects, reason codes, confidence bands, ambiguity, confirmation status.
- `ProjectAuditTimelineProjection`: metadata-only state changes and maintenance actions.
- `ConversationStartSetupProjection`: setup subset needed by Chatbot to start/resume a conversation.

Every projection should define:

- Owner.
- Source events.
- Projection key.
- Tenant scoping model.
- Rebuild behavior.
- Watermark/ETag behavior.
- Freshness semantics.
- Query authorization behavior.
- CLI/MCP/Web field parity.

Query responses should explicitly represent `included`, `excluded`, `unauthorized`, `unavailable`, `stale`, `archived`, `ambiguous`, `tenant_mismatch`, `conflict`, and `invalidReference` states where applicable. If the projection cannot prove safe inclusion, the response should fail closed.

### 6. Compliance and Data Protection Synthesis

The compliance posture is straightforward: **Projects should minimize data by design.** GDPR, CCPA/CPRA, FTC guidance, ISO/IEC 27001, NIST CSF, OWASP ASVS, and AI governance trends all favor least data, clear purpose, access control, auditability, and security evidence.

Projects should persist:

- Project identity, tenant, lifecycle, and safe display/setup metadata.
- Stable references to owning bounded contexts.
- Reference health and authorization/freshness statuses.
- Resolution reason codes and confirmation metadata.
- Audit identifiers and correlation metadata.

Projects should not persist:

- Transcript content.
- File contents.
- Memory payloads.
- Secrets, tokens, credentials, raw JWTs.
- Raw prompts or model responses.
- Unrestricted filesystem paths.
- Unsafe derived summaries.
- EventStore envelope metadata fields.

This approach makes event sourcing compatible with privacy expectations. Immutable audit trails become a strength because the stream contains safe metadata, not payload data that later requires selective erasure.

### 9. Risk Register and Mitigations

High-priority risks:

- **Payload leakage into immutable events.** Mitigate with event catalog, forbidden-field tests, code review checklist, and redaction-safe logging.
- **Cross-tenant query/projection leakage.** Mitigate with canonical identity derivation, tenant-scoped keys, authorization-filtered queries, and negative tests.
- **Projection staleness causing unsafe Chatbot context.** Mitigate with watermarks, ETags, freshness policies, and fail-closed context assembly.
- **Read model drift treated as source of truth.** Mitigate by routing writes only through EventStore commands and making projections rebuildable.
- **Event evolution breaking replay.** Mitigate with additive contracts, sample streams, replay tests, and compatibility gates.

Medium-priority risks:

- **Projection sprawl.** Mitigate with projection catalog ownership and rebuild requirements.
- **Telemetry cardinality.** Mitigate by using identifiers in traces/logs and limiting metric dimensions.
- **MCP over-orchestration.** Mitigate by keeping MCP as command/query translation, not domain workflow logic.
- **AI governance overclaim.** Mitigate by documenting Projects as metadata evidence provider, not full AI compliance authority.

The final recommendation is:
