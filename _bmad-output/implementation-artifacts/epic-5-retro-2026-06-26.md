## What Went Well

1. The metadata-only rule held across the epic.
   - Audit rows derive from Project events and EventStore metadata, not payloads.
   - Operator diagnostics intentionally omit idempotency keys and raw command/problem bodies.
   - Safe export, CLI output, MCP results, rendered markup, and test fixtures all preserve the no-payload boundary.

2. The "one operational model, three surfaces" goal became real.
   - Web views, MCP resources/tools, and CLI commands reuse the same vocabulary, action names, lifecycle names, reason codes, audit identifiers, and payload-exclusion guarantees.
   - Story 5.10 review caught MCP/CLI parity gaps and fixed warning/dashboard `diagnosticUnavailable`, lifecycle constants, and safe failure mapping.
   - Story 5.11 added final Web/MCP/CLI parity tests and cross-tenant denial hardening.

3. Reviews found material issues before closure.
   - Story 5.3 fixed a non-reproducible FrontComposer inspect gate.
   - Story 5.4 fixed transport exception handling so UI sources render safe feedback instead of crashing.
   - Story 5.6 fixed the policy-redacted trace classification bug.
   - Story 5.8 fixed dashboard tile drill-in mismatches.
   - Story 5.9 fixed relink target evidence and confirmation gating.
   - Story 5.10 fixed MCP denial/unavailable failures being swallowed as empty success.
   - Story 5.11 corrected initially over-claimed accessibility/cross-tenant coverage and replaced it with runnable checks.

4. The implementation reused existing seams rather than expanding the backend unnecessarily.
   - Audit timeline UI reads through `GetProjectOperatorDiagnosticsAsync` rather than adding a duplicate audit endpoint.
   - Reference health and warnings/dashboard use generated client queries and bounded enrichment rather than Web sibling fan-out.
   - Resolution trace remains compute-on-demand and transient; no trace projection or persisted score/rank history was added.
   - Maintenance actions reuse existing command-async routes where semantics fit.

5. Restore was implemented correctly as a domain capability.
   - Story 5.9 added `RestoreProject`, `ProjectRestored`, and `ProjectRestoreRejected` through Contracts, aggregate, projections, server endpoint, OpenAPI, generated client, audit mapping, docs, and tests.
   - The story avoided Web-only lifecycle overrides, direct read-model writes, and Dapr-state shortcuts.

## Challenges And Growth Areas

1. Several stories initially over-claimed validation.
   - VSTest, AppHost, Playwright, and E2E typecheck were sometimes marked complete before the actual runnable lane existed in the current environment.
   - Later reviews corrected those claims, either by running the gate, replacing blocked live-AppHost checks with no-AppHost contracts, or explicitly documenting the remaining blocker.
   - Lesson: story records should distinguish "specified command blocked", "compensating lane passed", and "actual required lane passed".

2. Surface parity defects appeared late.
   - MCP resource failure mapping returned empty success for downstream errors before Story 5.10 review fixed it.
   - CLI warnings/dashboard initially lacked the same enrichment and counters as Web/MCP.
   - MCP warning queue missed `DiagnosticUnavailable` until Story 5.11 review.
   - Lesson: parity cannot rely only on shared docs; it needs executable comparisons across Web selectors, MCP DTOs, CLI JSON, and parity-matrix fields.

3. FrontComposer customization needed explicit rationale.
   - Story 5.4 recorded a deliberate Level-4 escalation for inventory/detail because the required operational layout, tabs, selectors, and disabled filters were not expressible through generated bodies alone.
   - That was acceptable only because descriptor wrappers, leakage tests, parity docs, and inspect gates remained in place.
   - Lesson: when generated UI is not enough, the replacement still needs a descriptor-backed contract and a clear reason.

4. UI source failure handling had to be hardened repeatedly.
   - Story 5.4 added terminal safe exception mapping for inventory/detail/operator diagnostics.
   - Story 5.11 later normalized `401/403/404` to hidden-equivalent safe denial across Web sources.
   - Lesson: every generated-client wrapper should share the same failure taxonomy instead of reimplementing error mapping independently.

5. Some technical debt remains at the packaging and vocabulary edges.
   - Story 5.3 recorded that `Hexalith.Projects.Contracts` is packable but now carries Blazor/Fluxor/FluentUI dependencies through projection descriptor hosting.
   - Story 5.5 recorded mixed `FreshnessTrustState` vocabulary across reference/evaluation/conversation sources.
   - Story 5.11 fixed mobile table wrapping issues, but live AppHost browser coverage remains deferred.

## Key Lessons

1. Cross-surface parity needs executable evidence, not only shared descriptors.
2. Safe failure mapping is part of the contract; empty success is a dangerous denial/unavailable bug.
3. Generated-client and FrontComposer seams work when adapters stay thin and review catches shortcuts.
4. Feature-complete operational UI still needs descriptor-backed contracts when it uses hand-authored layout.
5. A story record must never blur blocked gates, compensating checks, and fully executed checks.
