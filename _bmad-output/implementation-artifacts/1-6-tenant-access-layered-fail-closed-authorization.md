## Story

As a **security-conscious platform operator**,
I want **tenant membership/status projected locally from Tenants events and a layered, fail-closed authorization chain enforced on every Projects operation**,
so that **cross-tenant access is impossible by construction and unverifiable evidence denies access** _(realizes UJ-4; NFR-1, NFR-3; AR-13, AR-19; FS-8)_.

This is **Implementation Sequence step 5** (architecture §Decision Impact Analysis) and **Epic-1 story group 1b** (epics.md §"Epic 1 decomposition"): the local `TenantAccessProjection` + the full layered fail-closed authorization chain that Story 1.4 explicitly **deferred** (1.4 shipped only a minimal claim-based tenant guard sufficient to prove fail-closed-on-missing-tenant + safe-denial 404). It builds on 1.1 (scaffold/CI), 1.2 (`ProjectId`/`ProjectIdentity` + shared `Ui/` vocabulary + `PayloadClassification`), 1.4 (`CreateProject` slice + `IProjectTenantContextAccessor` minimal guard + `ProjectsDomainServiceEndpoints` safe-denial 404 + FS-8 cross-tenant isolation test seed), and 1.5 (projection rebuild/replay + idempotency proof suite). It fills the **empty** landing folders the scaffold left: `src/Hexalith.Projects/Authorization/` (domain-core authz decisions), `src/Hexalith.Projects.Server/Authorization/`, `src/Hexalith.Projects.Server/Authentication/`, `src/Hexalith.Projects.Server/TenantAccess/`, and `src/Hexalith.Projects.Workers/Tenants/`.

**Scope discipline — this story is the tenant-access / layered-authz layer for the EXISTING `CreateProject` + minimal `GetProject` slice ONLY.** Defer to their own stories: `GetProject`/full detail read + `ListProjects` query surfaces and query-side result filtering on those surfaces (1.7 — this story builds the reusable query-side filtering *mechanism* and applies it to the existing create/minimal-get endpoints, but does NOT add new query endpoints); `UpdateProjectSetup`/`ArchiveProject` (1.8); Aspire/Dapr/Workers operational topology, dead-letter, durable-store *production binding* + runbooks (1.9 — this story defines the **durable-store interface** with an in-memory implementation for tests and wires the Workers `ITenantEventHandler<T>` subscription handlers, but does NOT stand up the Aspire topology or a production Redis/Dapr-state binding); the referenced-resource authorization layer via owning-context ACLs (Epic 2 — `IProjectConversationDirectory`/`IProjectFolderDirectory`/`IProjectMemoryDirectory`); real-Keycloak E2E runtime security proof (Epic 5 hardening — see Security Invariants for the in-scope vs deferred split). **This story owns the first FIVE layers of the AR-19 chain** (API/JWT → EventStore claim-transform → `TenantAccessProjection` exists/active/member/role → project-level authz belongs-to-tenant/accessible/active → query-side result filtering), the Dapr deny-by-default policy *evidence layer*, and the Workers tenant-event subscription that feeds the projection. The **referenced-resource-via-owning-context** layer is the only AR-19 layer deferred (to Epic 2).

## Acceptance Criteria

1. **Tenants events → idempotent `TenantAccessProjection` through a durable-store interface; not-yet-usable / fail-closed after restart (AR-13, NFR-3, NFR-7).**
   **Given** Tenants publishes lifecycle/membership events (`TenantCreated`, `TenantUpdated`, `TenantEnabled`, `TenantDisabled`, `UserAddedToTenant`, `UserRemovedFromTenant`, `UserRoleChanged`, `TenantConfigurationSet`, `TenantConfigurationRemoved`)
   **When** `Hexalith.Projects.Workers` consumes them via Dapr pub/sub (the `Hexalith.Tenants.Client` `ITenantEventHandler<T>` subscription) into the local `TenantAccessProjection`
   **Then** handlers are **idempotent (dedupe by message id)**, tolerate **out-of-order / at-least-once delivery** (a sequence ≤ the projection watermark is dropped, not re-applied; a same-message-id with a different payload fingerprint flags a **replay conflict** that fails closed), and write through a **durable-store interface** (`IProjectTenantAccessProjectionStore` — in-memory implementation for tests, production binding deferred to 1.9) with **optimistic-concurrency (version) writes**
   **And** after a restart the projection for a tenant with no replayed evidence is treated as **not-yet-usable** and the authorizer **fails closed** (an `UnknownTenant`/`UnavailableProjection`/`MalformedEvidence` outcome denies — never a silent allow) until the projection is rebuilt/usable.

2. **Layered, ordered, fail-closed authorization chain on every Projects operation (AR-19, NFR-1; OWASP API #1 object-level authz).**
   **Given** an authenticated request (JWT/OIDC, Keycloak realm `hexalith`)
   **When** authorization is evaluated for a Projects operation
   **Then** the chain runs **in declared order** and short-circuits on the **first denial**: **(1)** API/JWT validation → **(2)** EventStore claim-transform evidence (authoritative tenant + principal + permission tokens; absent/malformed evidence → deny) → **(3)** `TenantAccessProjection` (tenant exists / active / principal is a member / role) with freshness → **(4)** project-level authz (project belongs-to-tenant, accessible, lifecycle-active) → **(5)** query-side result filtering (a result set never contains an item outside the authoritative tenant)
   **And** **tenant authority is taken ONLY from authenticated claims + the EventStore claim-transform — never from request payload/header/query** (a client-supplied `tenantId` that disagrees with the authoritative tenant is a `TenantMismatch` denial, not an override)
   **And** Project data uses the **user-facing tenant as the EventStore envelope tenant** (NOT the reserved `system` platform tenant; a `system`-tenant context is never a valid project-data context and denies)
   **And** the ordered layer list + the effective-permission evidence sources are declared **once** as an enumeration (mirroring Folders' `AuthorizationOrder`), not re-spelled per call site.

3. **Missing / unknown / disabled / stale / unauthorized evidence → safe-denial 404 with a structured reason code; no existence leakage (NFR-1, NFR-3, AR-16).**
   **Given** missing / unknown / disabled / stale / malformed / unauthorized evidence at **any** layer
   **When** a trust-bearing operation is attempted
   **Then** it **fails closed** and the HTTP surface maps the denial to a **safe-denial `404`** (RFC 9457 ProblemDetails + Hexalith extensions) with a structured reason code drawn from the shared vocabulary — **unauthorized and nonexistent are externally indistinguishable** (status, body, and timing must not let a caller infer that a tenant/project exists in another tenant), `details` is **metadata-only** (a denial reason category/code, never the evidence values, claims body, token, or payload), and a fail-closed denial is **never** a generic `500`
   **And** the denial is logged with **structured metadata only** (tenant/project/correlation ids + reason code + the failed authorization layer — never tokens, claims bodies, payloads, or unrestricted paths; cross-checked against `PayloadClassification.ForbiddenContent`).

4. **Cross-tenant isolation conformance suite — re-runnable against every surface (FS-8, SM-3, NFR-1).**
   **Given** the cross-tenant isolation conformance suite (FS-8, SM-3) established in Story 1.4
   **When** a request scoped to tenant A targets tenant B's project/data (via the create command, the minimal `GetProject` read, the `TenantAccessProjection` lookup, and the query-side filter)
   **Then** it **never returns tenant B's data** — the negative-test suite proves "Project A / tenant B context never leaks" across the create path, the read path, the projection-membership path, and the query-side-filter path
   **And** the suite is structured as a **reusable conformance harness** (mirroring the FS-8 seed + the 1.5 `ProjectionRebuildConformance` pattern in `Hexalith.Projects.Testing`) so it is **re-run against every new surface** (Epic 2 references, Epic 3 context assembly, Epic 5 surfaces) without rewriting the assertions
   **And** mandatory negative-path coverage is present and genuinely asserts (not happy-path-only): cross-tenant denial, missing-claim deny, disabled-tenant deny, non-member deny, stale-projection deny (mutation budget vs diagnostic-read budget), unknown-tenant-after-restart deny, replay-conflict deny, and existence-non-inference (A and B denials are byte-identical at the boundary).

5. **No compiler setting weakened; boundaries preserved; build + filtered lane + gates green (NFR all, project-context.md).**
   All touched projects keep `net10.0` (`Contracts` netstandard2.0-safe where applicable), `Nullable=enable`, `ImplicitUsings=enable`, `TreatWarningsAsErrors=true` — **no** `NoWarn`/`#pragma`/`SuppressMessage`/nullable-disable to force green, and **no** weakening of authorization/tenant-validation/query-side-filtering to simplify tests (use Tenants/Testing + EventStore/Testing fakes/builders + synthetic JWT/claims). Boundary direction holds: the **pure authorization decision logic + the `TenantAccessProjection` apply/handler + the durable-store interface live in domain core `Hexalith.Projects` (Tier-1 pure — no Dapr/network/HTTP)**; the **Server** owns the claim-transform integration, the `IProjectTenantContextAccessor` claims-bearing accessor, the endpoint wiring, and the EventStore authorization-validator binding; the **Workers** host owns the `ITenantEventHandler<T>` Dapr-pub/sub subscription; `Client`/`Cli`/`Mcp` never reference domain authz types or Dapr. `dotnet build Hexalith.Projects.slnx` (0 warnings) + `tests/tools/run-filtered-tests.ps1` + `run-openapi-fingerprint-gate.ps1` + `run-frontcomposer-inspect-gate.ps1` all pass green. Any new package version is added to root `Directory.Packages.props` reusing the exact sibling-pinned version (never inline); the spine/generated `.g.cs` stay byte-stable (no contract change is required by this story — authz is host behavior, not a new wire schema; if a denial reason surfaces on the wire it reuses the existing `SafeAuthorizationDenial`/ProblemDetails shape). No sibling submodule pointer changes.

### Security invariants (NON-NEGOTIABLE — this is the security-sensitive story)
- **Fail-closed by default:** absence of an explicit grant = deny. No default-allow, no "open if claim missing", no silent-allow on a not-yet-rebuilt projection. Every unhandled branch in the authorizer returns a denial. The EventStore authorization validator defaults to **deny-all**; an "allowing" validator is an explicit dev/test opt-in only. [architecture.md#Authentication & Security L246-259, #Process Patterns L426, AR-19; NFR-3]
- **Safe-denial (existence non-inference):** unauthorized AND nonexistent both surface as **404** — externally indistinguishable in status, ProblemDetails body, headers, and **timing**. Cross-tenant existence must not be inferrable. A fail-closed denial is **never** a generic 500. RFC 9457 ProblemDetails + Hexalith extensions (`category`/`code`/`correlationId`/`retryable`/`clientAction`/`details.visibility`); `details` is **metadata-only** (denial reason category/code + failed-layer name — never evidence values, claims, tokens, or payloads). [architecture.md#Format Patterns L401-404, #Anti-patterns L470-471, AR-16; NFR-1]
- **Layered enforcement at EVERY layer (not a single gate):** auth → aggregate identity → state keys → projection keys → queries → pub/sub topics → SignalR groups → logs. This story enforces auth (JWT/claim-transform/tenant-access/project-level/query-side) at the API layer AND tenant authority at the write layer (`/process` + EventStore validator) — defense in depth (project-context.md: "test both defense layers: API/JWT authorization and domain/query-side filtering"). [NFR-1, AR-19; project-context.md#Critical Don't-Miss]
- **Tenant authority source:** ONLY authenticated claims + EventStore claim-transform — NEVER request payload/query/header. A client-supplied tenant that disagrees with the authoritative tenant is a `TenantMismatch` denial, not an override. Project data uses the **user-facing tenant as the EventStore envelope tenant** (NOT `system`); a `system`-tenant context is never a valid project-data context. [architecture.md#Process Patterns L423-425, AR-4, AR-19; project-context.md L58]
- **Rejections-as-events / fail-closed result, not exceptions:** an authz denial in the domain/command path is a fail-closed result / rejection event (`ProjectCreationRejected` with a shared `ReferenceState` reason from Story 1.4), not a thrown exception; infrastructure failures (store unavailable) are the exception/dead-letter path but still **fail closed** at the authorizer (`UnavailableProjection` → deny). [AR-3, FS-4; project-context.md#Critical Don't-Miss]
- **Persist-then-publish; Dapr-only infra; reuse Story 1.2 reason codes (no parallel enum); metadata-only logging** (no tokens/claims-bodies/payloads/paths — cross-check `PayloadClassification`). [architecture.md#Communication/Process Patterns; NFR-2, AR-18, AR-20]
- **Additive / serialization-tolerant — no `V2` events.** The consumed Tenants events are owned by Tenants (do not fork them); the local projection + normalized internal event are additive. [NFR-6, AR-6]
- **In-scope vs deferred security proof:** synthetic JWT/claims are acceptable at the **unit (Tier-1) and integration (Tier-2)** tiers for this story (project-context.md: "synthetic JWT generators are only for fast unit/integration tests"). **Real-Keycloak/OIDC E2E is the EventStore runtime-security-proof convention** and is **deferred** (Epic 5 hardening / the e2e workspace) — this story proves the fail-closed *logic* and the layered *ordering* deterministically; it does NOT stand up a real Keycloak realm. Flag this split in the Dev Agent Record.

Fixed by authorizing first; invalid freshness is now surfaced only to authorized callers.

Fixed by returning internal `ReferenceState.TenantMismatch` / `tenant_mismatch` while preserving the external safe-denial envelope.
- **MEDIUM — Projects claim normalization did not promote standard EventStore-style permission evidence.** `ProjectsClaimsTransformation` copied tenant/principal evidence but not `permissions` → `eventstore:permission`, so valid EventStore-shaped JWT evidence could fail closed unnecessarily. Fixed with EventStore-style tenant/permission claim splitting and tests.

Fixed by failing closed as malformed authorization evidence before project ACL/EventStore/Dapr layers.
