# Story 2.4: Set & auto-create Project Folder

## Status

done

## Story

As Hexalith.Chatbot, I want to set the single authorized Project Folder for a Project, and to have one auto-created with the Project name when none is supplied at creation, so that a Project always has exactly one folder boundary without forcing the user to pre-create it (FR-8; completes deferred FR-1 auto-folder; realizes UJ-2).

This story is the Projects-owned Project Folder reference slice. Hexalith.Folders remains the system of record for folder lifecycle and authorization; Hexalith.Projects stores only the authorized folder reference and safe metadata.

Current Folders capability is contract-ready but not fully externally mapped: `Hexalith.Folders.Client.Generated.IClient.CreateFolderAsync(...)` and the Folders OpenAPI `POST /api/v1/folders` operation exist, but the current Folders server endpoint mapping does not expose `POST /api/v1/folders` as an external REST route. Development must either use an explicitly approved in-topology typed-client/Dapr/internal path for `CreateFolder`, or implement the named degraded behavior `ProjectFolderCreationPending` as a queued/flagged state that does not block project creation. If neither path is accepted, HALT before implementation.

## Acceptance Criteria

1. Projects exposes an authorized Set Project Folder mutation for an existing Project. The mutation is command-async, requires `Idempotency-Key`, preserves correlation/task headers, validates route/body Project identity equality, uses a closed request schema with `requestSchemaVersion`, and requires explicit replacement intent when replacing an existing Project Folder.
2. `SetProjectFolder` results in a metadata-only `ProjectFolderSet` event that records the stable Folder identity and safe Folder metadata. It never stores file contents, unrestricted paths, tenant payloads, repository internals, or upstream raw authorization details.
3. A Project has exactly one Project Folder while active. The folder may be replaced through the explicit mutation, but active Project Folder removal is rejected. If an archived-only removal surface is introduced, it must be explicitly guarded by archived lifecycle state and authorization.
4. Folder authorization and lifecycle validation are delegated to Hexalith.Folders through a Projects server ACL, for example `IProjectFolderDirectory`. Projects does not decide folder access from local payloads and treats denied, missing, stale, archived, or unavailable folder evidence as fail-closed.
5. When a Project is created without a supplied folder, Projects requests creation of a Project Folder using the Project name as the folder display name. The request uses the Folders `CreateFolder` contract, stable idempotency for the Project/folder-create attempt, and correlation/task propagation. Projects must not assume read-after-write; it confirms via Folders lifecycle/freshness evidence before recording `ProjectFolderSet`.
6. Because the external Folders create endpoint is not currently server-mapped, implementation must choose and document one of two paths in the Dev Agent Record before coding the auto-create lane:
   - Available API path: use an approved in-topology path to the contract-ready Folders `CreateFolderAsync` capability, then confirm via lifecycle/freshness before `ProjectFolderSet`.
   - Degraded path: record/queue the metadata-only `ProjectFolderCreationPending` behavior, surface the folder reference as pending/unavailable using shared reference state, and keep project creation accepted. The pending record must include only safe identifiers, display-name intent, idempotency/correlation/task metadata, and reason code such as `folder_create_external_unavailable`; it must be retryable/reconcilable when `CreateFolder` becomes available.
7. Folders failure or denial maps to a Projects-safe problem result with shared reason/state codes. Raw upstream details, stack traces, ACL payloads, and tenant authority are never exposed to clients, events, logs, or projections.
8. `ProjectReferenceIndexProjection` is established or updated for the Project Folder reference. Project read models and reference summaries expose the folder reference state without owning folder contents or bypassing Folders authorization.
9. Idempotency behavior is field-scoped and deterministic. `SetProjectFolder` duplicate-equivalent requests replay or no-op safely; conflicting repeats reject. If `CreateProject` accepts an optional supplied folder or auto-create intent, create idempotency equivalence is updated to include those folder-selection fields.
10. Public contract changes are made through the OpenAPI/API spine and generated artifacts are regenerated by the existing project workflow. Generated `.g.cs` files are not hand-edited.
11. The implementation is covered by focused aggregate/state, projection, endpoint, ACL adapter, idempotency, auto-create/degraded-path, no-payload-leakage, and contract/client generation tests.

## Dev Notes

### Current Implementation Facts

- Projects currently has no Project Folder state in `ProjectState`, no `ProjectFolderSet` event catalog entry, and no Set Project Folder endpoint.
- The Projects OpenAPI currently documents project create/list/get, setup/archive, and Story 2.3 conversation reference endpoints. Project create still describes the Project Folder as deferred.
- Existing Projects reference modeling uses string sibling references rather than local value objects for sibling aggregate IDs.
- Folders OpenAPI contains `POST /api/v1/folders` with operationId `CreateFolder`, and the generated Folders client exposes `CreateFolderAsync(...)`.
- Current Folders server mapping exposes internal `/process`, project metadata, effective permissions, archive, and lifecycle status routes, but not external `POST /api/v1/folders`.
- Folders command mutations are asynchronous; successful create acceptance is not proof that the folder lifecycle is fresh or active.

### Guardrails

- Keep the aggregate pure. Do not call Folders, Dapr, HTTP clients, or other external services from aggregate handlers, state apply code, or projections.
- Treat Hexalith.Folders as the lifecycle and authorization owner. Projects owns only the Project-to-Folder reference and metadata needed for Projects UX/contracts.
- Store folder identifiers as sibling reference strings according to the identifier-boundary ADR; do not introduce a Projects-owned `FolderId` value object.
- Never store or echo folder contents, unrestricted paths, repository internals, tenant authority, or raw upstream denial details.
- Preserve Pattern A reference ownership: sibling contexts own mutable assignment and authorization for their resources.
- Do not hand-edit generated client files. Change the OpenAPI/API spine and regenerate through the established workflow.
- Do not initialize or update nested submodules recursively.

### Suggested API Shape

The exact route may follow local endpoint conventions, but the intended behavior is:

```http
PUT /api/v1/projects/{projectId}/folder
Idempotency-Key: <required>
X-Correlation-Id: <propagated>
X-Hexalith-Task-Id: <propagated>
```

Request body:

```json
{
  "requestSchemaVersion": "1.0",
  "projectId": "proj_...",
  "folderId": "fold_...",
  "folderMetadata": {
    "displayName": "Project name"
  },
  "replacementConfirmed": true
}
```

For create-time auto-folder behavior, an existing `CreateProject` request without a supplied folder triggers the auto-create lane. If an explicit supplied folder field is added to `CreateProjectRequest`, it must be additive, closed-schema, and included in create idempotency equivalence.

### Folders Capability Decision Required

The story must not silently assume external `CreateFolder` REST availability.

Available local contract:

- `Hexalith.Folders.Client.Generated.IClient.CreateFolderAsync(...)`
- Folders OpenAPI `POST /api/v1/folders`

Current server gap:

- External Folders `POST /api/v1/folders` is not mapped in the current Folders server endpoint registration.

Required decision:

- Use an explicitly approved in-topology typed-client/Dapr/internal route to the Folders create capability, or
- Use degraded queued/flagged behavior named `ProjectFolderCreationPending`.

`ProjectFolderCreationPending` means Project creation remains accepted, the pending folder create intent is stored or queued with metadata-only identifiers and retry metadata, Project reads show folder state as pending/unavailable, and `ProjectFolderSet` is not recorded until Folders lifecycle/freshness evidence confirms the created folder.
