# Story 1.3: OpenAPI Contract Spine + generated typed client

Status: done

## Story

As a **Chatbot integration engineer**,
I want **the OpenAPI 3.1 Contract Spine to be the single source of truth driving a generated typed client, with idempotency, correlation, freshness headers, and RFC 9457 errors**,
so that **consumers call a stable, version-checked contract and mutations/queries behave predictably (command-async, safe-denial)**.

This is **Implementation Sequence step 2 (tail)** — the OpenAPI spine that finishes the "Contracts-before-code" foundation (AR-2/AR-15). It builds on Story 1.1's empty `Contracts/openapi/` + `Client/` skeleton and Story 1.2's `ProjectId` + shared vocabulary. It is **the contract spine + the generated typed client + the idempotency hasher + the fingerprint/staleness gates flipping ON** — NOT the aggregate, projections, server endpoints, ACLs, or FrontComposer surfaces (those land in 1.4+ / Epic 5). The spine defines shared contract *language* and the cross-cutting envelope (headers, RFC 9457 errors, command-async, safe-denial) now; concrete operation groups (CreateProject, GetProject, ListProjects, …) are added by their own stories.

## Acceptance Criteria

1. **OpenAPI 3.1 Contract Spine is the single source of truth (AR-15).** `src/Hexalith.Projects.Contracts/openapi/hexalith.projects.v1.yaml` is authored as a valid **OpenAPI 3.1.0** document. It declares: `info` (title `Hexalith.Projects API`, version `v1`), a URL-versioned `/api/v1` server, the OIDC bearer security scheme applied globally, the reusable **header parameters** `Idempotency-Key` / `X-Correlation-Id` / `X-Hexalith-Task-Id`, the reusable **response headers** `X-Correlation-Id` / `X-Hexalith-Task-Id` / `X-Hexalith-Freshness`, and the reusable **components** for the cross-cutting envelope: `AcceptedCommand` (202 body), the freshness/trust metadata schema, the `ProblemDetails` schema (see AC-3) plus its safe-denial / validation / idempotency-conflict response wrappers. JSON wire shape is **camelCase**; dates are **ISO-8601 `DateTimeOffset`**. The document is `additionalProperties: false` on closed envelope schemas and carries synthetic, payload-free examples only (no secrets, no transcript/file/memory content, no real tokens, no machine-local paths). **At least one representative operation per shape is wired so the spine drives real generated output** (e.g. one command-async `POST` returning `202 AcceptedCommand` and one freshness-carrying `GET` query) — but the full Project operation catalog is explicitly deferred to the command/query stories (1.4/1.7/1.8).

2. **NSwag-generated typed client lives under `Client/Generated/*.g.cs` and is never hand-edited (AR-15, AR-17).** `src/Hexalith.Projects.Client/nswag.json` is authored to consume **only** the spine at `../../src/Hexalith.Projects.Contracts/openapi/hexalith.projects.v1.yaml` as its input, emitting `Generated/HexalithProjectsClient.g.cs` in namespace `Hexalith.Projects.Client.Generated`. An MSBuild `BeforeTargets="BeforeCompile"` target regenerates the client from `(spine, nswag.json)` inputs so the checked-in `.g.cs` is reproducible, never hand-authored. The Client project also exposes `AddProjectsClient(...)` DI registration (typed `HttpClient`, returns `IHttpClientBuilder` so callers attach a bearer `DelegatingHandler`; auth strategy stays out of the Client) and `ProjectsClientOptions` (absolute `BaseAddress`, fail-fast validation). Generated artifacts contain **no** machine-local absolute paths, no `https://` host literals, and no secret/token strings.

3. **Idempotency hasher + field-scoped equivalence (AR-15, AR-16).** A `Hexalith.Projects.Client.Idempotency.HexalithIdempotencyHasher` produces a deterministic, culture-invariant `sha256:<hex>` over a canonical `(operationId, ordered fields)` representation, mirroring the Folders hasher contract (ordinal-ascending declared field order, escape rules, type-stable canonicalization, UTC-only date handling). Each mutation operation in the spine declares its `x-hexalith-idempotency-equivalence` field list, and the generated idempotency helpers (`Generated/HexalithProjectsIdempotencyHelpers.g.cs`) expose `ComputeIdempotencyHash(...)` per request schema, generated from the spine — never hand-edited. The spine documents `Idempotency-Key` as **required on mutations and rejected on queries** (`x-hexalith-idempotency-*` annotations), and `X-Correlation-Id` / `X-Hexalith-Task-Id` are threaded.

4. **Command-async API shape + freshness/trust (AR-16).** Every mutation operation in the spine returns **`202 AcceptedCommand`** (the contract documents **no read-after-write guarantee**); query operations carry **freshness/trust state** (`X-Hexalith-Freshness` response header + freshness metadata in the body schema). The 202 body is the closed `AcceptedCommand` schema (`acceptedAt`, `correlationId`, `taskId`, `status: accepted`, optional `idempotentReplay`).

5. **RFC 9457 Problem Details + Hexalith extensions + safe-denial (AR-16, NFR-1, NFR-2).** Error/denial responses use `application/problem+json` with an RFC 9457 `ProblemDetails` schema carrying the Hexalith extensions: `category`, `code`, `correlationId`, `retryable`, `clientAction`, and `details.visibility`. **Safe-denial: unauthorized and nonexistent both surface as `404`** (externally indistinguishable) so cross-tenant existence cannot be inferred — the spine wires a shared `SafeAuthorizationDenial` (401/403/404) response set on protected operations. **No error echoes secrets, payloads, or sensitive values** — `details` is metadata-only and the spine's examples contain no forbidden content per the Story 1.2 payload taxonomy. Reason codes surfaced in errors draw from the shared vocabulary (Story 1.2 enums), never free-text-as-canonical-signal or a new parallel enum.

6. **Fingerprint / contract-stability gate runs in CI and fails on incompatible drift (AR-15, AR-17, FS-7).** A generated provenance artifact records the SHA-256 of `(spine, nswag.json, generated helpers)` (normalized line endings), and a verification API (`HexalithProjectsGeneratedArtifacts.VerifyCurrentDetailed(repositoryRoot)`) returns false with a path-relativized diagnostic when checked-in generated output drifts from the spine inputs. The two Story-1.1 input-gated gate scripts are **flipped from no-op-clean to real gates now that inputs exist**: `tests/tools/run-openapi-fingerprint-gate.ps1` runs the real fingerprint/compatibility check (drift → fail), and `tests/tools/run-frontcomposer-inspect-gate.ps1` runs the real `frontcomposer inspect --fail-on-warning` **if and only if** `[Projection]`/`[Command]` annotated contracts exist (this story authors the OpenAPI spine, not FrontComposer annotations, so this gate may legitimately remain in skip-clean mode unless the dev adds annotations — see Dev Notes). Replacing a skip branch with `exit 1`-placeholder-only is **not** acceptable; the gate must execute the real check. Tier-1/Contracts + Client tests assert: spine validity, nswag.json uses the spine as its only input, generated provenance is stable & safe (no absolute paths / hosts / tokens), the fingerprint detects drift, and the hasher is deterministic.

7. **No compiler setting weakened; boundaries preserved; build + filtered lane green.** All touched projects keep `net10.0` (`Contracts` netstandard2.0-safe), `Nullable=enable`, `ImplicitUsings=enable`, `TreatWarningsAsErrors=true` — no `NoWarn`/`#pragma`/`SuppressMessage`/nullable-disable to force green. `Contracts` stays low-dependency (the spine is a YAML asset + reuse of Story 1.2 types; no new infrastructure deps). `Client` may take **only** the generator/serialization deps the Folders sibling uses (`Newtonsoft.Json`, `NSwag.MSBuild`, the YAML helper for the idempotency generator) — **no Dapr, no EventStore.Server, no domain event types** in `Client`. `dotnet build Hexalith.Projects.slnx` and the filtered Tier-1 + Contracts + Client test lane pass green with zero warnings. Central package management only — any new package version is added to root `Directory.Packages.props` reusing the exact sibling-pinned version, never inline.
