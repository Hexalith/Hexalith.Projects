## Successes

1. Contracts-first sequencing reduced drift.
   - Story 1.2 established shared identity, vocabulary, rejection taxonomy, and payload classification before aggregate/API work.
   - Story 1.3 made the OpenAPI spine and generated client authoritative before later API expansion.
   - Stories 1.7 and 1.8 grew the contract through the spine and regenerated clients rather than hand-editing generated artifacts.

2. High-risk invariants received executable proof.
   - Story 1.4 added no-payload leakage and schema-evolution corpus coverage.
   - Story 1.5 added projection rebuild/replay/idempotency conformance.
   - Story 1.6 added the fail-closed tenant authorization chain and cross-tenant negative matrix.
   - Story 1.9 added durable Dapr-backed projection journal and topology tests.

3. Review cycles caught material defects before completion.
   - Story 1.6 review fixed safe-denial leakage, authorization ordering, claim normalization, and projection-tenant consistency.
   - Story 1.7 review added direct list authorization proof and cleaned generated-client whitespace.
   - Story 1.8 review fixed setup idempotency parity, process payload schema enforcement, validator coverage, and projection conformance coverage.
   - Story 1.9 review fixed tenant-wide watermarking, identity mismatch handling, replay fingerprinting, missing/conflicted journal behavior, and runtime evidence provider registration.

4. Local operational skeleton landed without moving sibling submodule pointers.
   - AppHost, ServiceDefaults, Server, Workers, Dapr config, Redis-backed local components, dead-letter topics, runtime adapters, and runbook documentation were added.
   - The story records explicitly report no recursive submodule churn.

5. Gate behavior became more honest as inputs landed.
   - Initial skip-clean gates were input-presence based, not unconditional.
   - OpenAPI fingerprint gates activated once the spine landed.
   - FrontComposer remained skip-clean because no `[Projection]` or `[Command]` contracts were introduced.

## Challenges And Growth Areas

1. Generated artifacts and line endings repeatedly required attention.
   - Story 1.1 review fixed LF-to-CRLF issues.
   - Story 1.3 documented generated `.g.cs` LF behavior as an intentional exception.
   - Story 1.7 review found generated client trailing whitespace.
   - Story 1.8 and Story 1.9 verification reported Git CRLF normalization warnings without whitespace failures.

2. Authorization ordering and safe-denial behavior needed adversarial review.
   - Story 1.6 review found that HTTP denial envelopes, GET freshness validation, claim evidence, and tenant consistency needed tightening.
   - This is a process lesson: every new read/write surface in Epic 2 needs direct authorization-gate tests plus endpoint behavior tests.

3. Idempotency equivalence is easy to drift when generated helpers and aggregate fingerprints evolve separately.
   - Story 1.8 review found setup-update idempotency parity drift between generated helper behavior and aggregate fingerprinting.
   - Future mutation stories should include direct equivalence tests crossing spine metadata, generated helper output, and aggregate validation.

4. Durable projection semantics required more precision than the story draft alone provided.
   - Story 1.9 review found per-aggregate sequence numbers were unsafe as tenant-wide watermarks.
   - Event identity, EventStore metadata, replay fingerprints, and fail-closed journal state all need to be first-class design inputs for future projection stores.

5. Some readiness facts are outside repository evidence.
   - Production deployment is not verified in these artifacts.
   - Stakeholder acceptance is not verified in these artifacts.
   - Real Keycloak/OIDC E2E remains deferred.
   - Aspire doctor passed with HTTPS development certificate warnings only; local certificate cleanup remains environment hygiene, not an Epic 1 blocker.

## Key Insights

1. "Mirror proven sibling patterns" worked, but only when the story also forced variance documentation.
   - Examples: Folders contract spine, EventStore command pipeline, Tenants tenant-access projection, and NSwag generation.
   - Variance documentation prevented silent divergence around test naming, SDK choices, generated line endings, and identity boundary decisions.

2. The strongest Epic 1 stories combined vertical functionality with reusable conformance hooks.
   - The no-payload leakage harness, projection rebuild conformance, schema corpus, and cross-tenant matrix now form a reusable foundation for later epics.

3. The code review loop materially improved the architecture.
   - Several review fixes were not cosmetic. They changed security, idempotency, durable projection, and runtime fail-closed behavior.

4. Story status alone is not enough for readiness.
   - The sprint status proves story completion.
   - Deployment, stakeholder acceptance, and local AppHost runtime behavior beyond tests still require external validation before treating the epic as released.

## Next Epic Preview

Epic 2 is "Context References." It makes a Project a workspace boundary by connecting it to conversations, a single Project Folder, optional file references, and memories through Projects-owned anti-corruption layers. Underlying resources stay owned by their contexts; unlinking does not delete them.

Epic 2 depends directly on Epic 1 foundations:

- Tenant-safe project identity and authorization.
- Open/List read surfaces and trust/freshness semantics.
- Metadata-only payload taxonomy and no-payload leakage proof.
- Projection rebuild and duplicate-delivery conformance.
- Dapr/Aspire operational skeleton and worker topology.
- Contract spine and generated client discipline.

Epic 2 also carries dependency decisions that must be explicit before implementation:

- Story 2.1 must choose the Conversation reference read path: additive `IConversationClient.ListConversationsAsync` or `ConversationReadApi` HTTP call.
- Story 2.2 is an upstream capability in `Hexalith.Conversations` and must remain separately tested and committed.
- Story 2.4 depends on the Folders create-folder external capability or an explicit degraded path.
- Story 2.6 must settle whether Memories linkage targets a Case or MemoryUnit references.

No Epic 1 discovery fundamentally invalidates the Epic 2 plan. The plan remains sound, but Epic 2 has external dependency decisions that should be resolved in the relevant story records before code changes begin.
