## What Went Well

1. The compute-on-demand boundary held.
   - Story 4.1 created a pure `ProjectResolutionEngine` with no EventStore, Dapr, command, projection, `HttpClient`, or wall-clock dependency.
   - Story 4.2 and Story 4.3 acted as host adapters: they fetched safe evidence, mapped it to Projects-shaped records, and delegated scoring/outcome to the engine.
   - Proposal preview in Story 4.5 recomputed resolution and wrote no event/projection/state/trace.

2. Ambiguity stayed explicit.
   - The engine returns `NoMatch`, `SingleCandidate`, or `MultipleCandidates`.
   - Two or more qualifying candidates never collapse into a silent attach.
   - Story 4.4 requires `confirmed: true`, `resolutionResult: MultipleCandidates`, candidate evidence containing the selected Project, and an explicit assignment through Conversations before submitting the Projects command.

3. Pattern A ownership held.
   - Projects did not add local conversation membership.
   - Confirmation and proposal confirmation route through `IProjectConversationAssignmentDirectory` / Conversations reassignment.
   - `ProjectResolutionConfirmed` records only confirmed-choice metadata and not conversation ownership state.

4. Review cycles caught material issues before closure.
   - Story 4.1 corrected a real `CS1739` named-argument blocker and stale File List/test-count claims.
   - Story 4.2 corrected misleading safe-denial documentation and added direct ACL adapter coverage.
   - Story 4.3 found and fixed a critical `ProjectListProjection` desync that would have made list/reverse-reference reads throw after file or memory link events.
   - Story 4.4 added missing serialization/leakage/idempotency surface tests.
   - Story 4.5 fixed authorization-before-ledger ordering, fail-closed preflight exception handling, and server/client idempotency field-name lockstep.

5. The OpenAPI spine stayed load-bearing.
   - Query operations reject `Idempotency-Key` and do not gain generated idempotency helpers.
   - Mutations include declared equivalence fields and generated helper coverage.
   - The Story 4.5 `folder.folder_id` mismatch showed why generated-client/server lockstep tests matter.

## Challenges And Growth Areas

1. The first story's validation diagnosis was initially wrong.
   - Story 4.1 first attributed build failure to Linux restore/project-reference evaluation.
   - The actual blocker was a two-site named-argument mismatch in `ProjectResolutionEngine.cs`.
   - Lesson: when MSBuild reports a non-specific failure, re-run with enough output to locate the first compiler error before recording an environment diagnosis.

2. New read-model paths need direct coverage from the start.
   - Story 4.2 shipped a new ACL adapter without direct tests until review.
   - Story 4.3 shipped a reverse reference-index stack whose endpoint tests used stubs until review added the real read-model coverage.
   - Lesson: any new adapter/read-model stack needs at least one direct test that exercises the production mapper and projection chain, not only endpoint stubs.

3. Shared tenant journals mean every projection must tolerate every Project event it may replay.
   - Story 4.3 exposed that `ProjectListProjection` threw on file/memory link/unlink events.
   - Story 4.4 added `ProjectResolutionConfirmed`, and the projections now explicitly tolerate it.
   - Lesson: adding a new Project event requires projection-catalog and projection-test updates for list, detail, and reference-index behavior, even when the projection stores no new visible field.

4. Composite mutation idempotency has more than one boundary.
   - Story 4.4 coordinates Conversations assignment and Projects command submission.
   - Story 4.5 coordinates create, assignment, folder set, and file link child operations under one root key.
   - The root ledger must run after authorization/preflight but before the first write; child keys must be deterministic; same-key/different-body conflict must be explicit.

5. Documentation drift remained after implementation until the retrospective audit.
   - Architecture still described Conversations assignment as an open blocker.
   - Projection docs omitted `ProjectResolutionConfirmed` and the Story 4.3 list-projection tolerance fix.
   - The shared vocabulary tracker still treated Epic 4 resolution producers as future.
   - The negative-test checklist under-described mutation-side retryable preflight failures.

6. Submodule pointer drift is still a process risk.
   - Story 4.5 review identified an unrelated `Hexalith.Folders` pointer move not required by the story.
   - It was not auto-reverted because submodule relationship changes are deliberate workspace decisions.
   - Lesson: submodule pointer diffs need an explicit gate before story closure, separate from ordinary file-list bookkeeping.

7. A pre-existing CreateProject `metadataClass` enforcement gap surfaced through Story 4.5.
   - Story 4.5 confirm now enforces `metadataClass` for proposal confirmation.
   - The pre-existing `CreateProject` endpoint still has the same OpenAPI-required-field enforcement gap.
   - This is outside Epic 4's scope but affects API/spec consistency.
