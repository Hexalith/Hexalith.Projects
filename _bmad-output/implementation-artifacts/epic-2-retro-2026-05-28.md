## Challenges And Growth Areas

1. NSwag idempotency-helper generation has a recurring cross-platform tooling bug.
   - Stories 2.3, 2.4, 2.5, and 2.7 all hit the same Windows-backslash MSBuild path that fails on Linux, requiring manual forward-slash invocation.
   - `.g.cs` was never hand-edited, but the manual regeneration step is a per-story friction the team should fix in tooling rather than working around four times.
   - Story 2.4 review made an improvement: the NSwag client target now normalises generated output as part of generation instead of expecting manual `.g.cs` cleanup. The Linux-path issue is distinct from the cleanup issue and remains.

2. The shared idempotency canonicaliser from Story 1.3 has a small but pre-existing escape-divergence between client helper and server hasher for `U+2028` / `U+2029`.
   - Story 2.5 senior review documented this as a LOW observation, out of Story 2.5 scope.
   - Story 2.7 inherits the same property for memory fingerprints.
   - Because cross-surface fingerprint parity is a load-bearing security/idempotency invariant, this debt should not remain LOW indefinitely; if any input ever includes those line-separator code points, the helper and server will compute different hashes and a duplicate command could be accepted as a fresh command.

3. Trust-boundary identifiers needed explicit architecture decisions mid-implementation.
   - Story 2.3 HALTed because the existing principal/JWT pipeline did not surface a stable Party id usable as `ConversationCommandMetadata.ActorPartyId`. `DeterministicActorPartyResolver` resolved the HALT but is explicitly documented as a non-reversible integration mapping waiting for a Parties-backed lookup.
   - This is the second consecutive epic where an "authorization shape" decision had to be settled adversarially during a story (Story 1.6 in Epic 1 — safe-denial leakage, claim normalization, projection tenant consistency). The pattern is: trust boundaries reveal themselves under load, not under planning.
   - Future write-side stories should treat "what is the canonical actor/party identity" as a Day-0 Dev-Agent question, not a Day-1 obstacle.

4. Capability-gate degraded paths leave reconciliation work for later.
   - `ProjectFolderCreationPending` (Story 2.4) is a queued/flagged state; when Folders external `POST /api/v1/folders` is mapped (Folders-submodule work, out of Projects scope), Projects needs a retry/reconciliation flow.
   - The current contract is intentionally fail-closed-clean: pending state is durably recorded as metadata-only with reason code `folder_create_external_unavailable`, and the Projects user can re-attempt set folder. This is acceptable, but the reconciliation is not yet a tracked Epic 5 or Epic 3 story.

5. Line-ending discipline still required explicit attention.
   - Story 2.6 (an ADR-only deliverable) caused whole-file CRLF/LF churn in `docs/adr/identifier-boundary.md` and `_bmad-output/planning-artifacts/epics.md`, which had to be re-normalised by the senior review.
   - Story 2.6 produced a build-environment memory entry codifying the LF-on-disk rule for `.md`.
   - This is the second consecutive epic with line-ending churn (Story 1.1 fix in Epic 1, Story 2.6 fix in Epic 2). It is now a recurring quality theme.

6. Some review feedback themes repeated across stories.
   - **Unsafe identifier echo in rejection events** — Story 2.4 had a real leak auto-fixed (folder reference id was being echoed in `ProjectReferenceLinkRejected`); Story 2.5 leakage tests now explicitly cover this taxonomy and similar checks were preventative in Story 2.7.
   - **Route/body identity validation** — Stories 2.1, 2.2, 2.3, 2.4, 2.5 each required negative tests for malformed body identity, route/body mismatches, and missing `Idempotency-Key`. The pattern is well-established now and should become a code-review checklist item.
   - **AC-exhaustiveness gaps in failure-mapping tables** — Story 2.6 senior review caught two HIGH gaps in the failure-to-shared-vocabulary table (`TenantStatus`, `MEMORY_UNIT_NOT_FOUND`); future ADR-only stories should run the same AC matrix check that mutation stories use.

7. "Unproduced" outcomes are accumulating in the shared vocabulary.
   - `TenantMismatch` and `Stale` outcomes are defined in `ProjectFileReferenceValidationOutcome` / `ProjectMemoryValidationOutcome` for taxonomy symmetry, but neither is currently produced by any ACL (Stories 2.5 and 2.7 both note this).
   - Carrying unproduced outcomes is acceptable when they have a planned producer (Folders Stale states are real); the team should periodically reconcile so the shared vocabulary stays an honest map of the world.

8. Some readiness facts remain outside repository evidence.
   - Stakeholder acceptance is not verified in artifacts.
   - Production deployment is not verified in artifacts.
   - Real Keycloak/OIDC E2E remains deferred (planned for Epic 5).
   - Local AppHost smoke check from Epic 1 action items did not produce a documented evidence trail in Epic 2 stories — the Aspire integration tests (14/14 stable across Stories 2.4–2.7) are the substitute proxy.

## Key Insights

1. ADR-as-deliverable scales.
   - Story 2.2 (upstream Conversations ADR) and Story 2.6 (Memories link-target ADR) showed that an enabler story can deliver a single durable artifact that the consuming story (Story 2.3, Story 2.7) executes against without re-deciding anything.
   - The Story 2.7 implementation explicitly references the Story 2.6 ADR as "decision locked in," which is the intent.
   - Future epics should continue this pattern wherever a decision spike precedes a write-side story.

2. Story 2.5's optional-reference shape is now the template.
   - Story 2.7 mirrored Story 2.5 almost verbatim: per-kind reference-index lane, fail-closed validation outcomes, "unlink takes no ACL dependency" structural proof, "link never reads content" structural proof, leakage harness extension, cross-surface fingerprint parity.
   - The reused shape compressed the Memory link/unlink design to almost no new design surface. Other Epic 2 stories were comparatively variable; this one converged.

3. Capability gates are now a first-class control flow construct.
   - Every story that depends on an external surface now starts with a documented gate decision (available / in-topology / degraded) rather than an assumption.
   - This converted what would have been multiple mid-story HALTs in earlier epics into pre-flight choices.
   - Story 2.3's HALT (for `ActorPartyId`) is the exception that proves the rule: a trust-boundary surface inside the umbrella behaved like an external dependency and needed the same gate discipline applied retroactively.

4. The boundary between "Projects-owned ACL" and "owning context" stayed sharp under pressure.
   - No domain or projection code references `Hexalith.Conversations.*`, `Hexalith.Folders.*`, or `Hexalith.Memories.*` types.
   - Story 2.7 enforced the Memories namespace boundary structurally via a test grep.
   - Story 2.7 used central NuGet management to access Memories without taking a submodule pointer dependency — a measurable architectural choice in favor of containment.

5. Fail-closed posture extends from authorization to availability.
   - Story 2.1 review established the "scope-poisoning closes the whole page" rule.
   - Story 2.4 added "degraded folder creation never blocks project creation but is never silently treated as available."
   - Story 2.7 added "absent or unavailable Memories reference is excluded with a fail-closed-clean reason code from the existing shared vocabulary — never a 5xx."
   - Pattern: when in doubt about evidence, exclude with a reason code; never silently include or silently fail.

## Next Epic Preview

Epic 3 is "Project Context Assembly." It uses the ACLs and reference index built in Epic 2 to compose a tenant-scoped, authorization-filtered, allowlist-assembled `ProjectContext` — setup plus included references with explicit exclusion reasons — plus an explanation of inclusion/exclusion, a refresh path that surfaces degraded references, and the conversation-start setup subset. Assembly is allowlist-based and fails closed; everything is metadata-only.

Epic 3 depends directly on Epic 2 outputs:

- The four ACLs (Conversations, Folders, Memories, Tenants) and the `Unavailable*Directory` fail-closed fallbacks.
- `ProjectReferenceIndexProjection` per-kind lanes (`folder` / `file` / `memory`) and the conversation read ACL (Pattern A).
- Shared vocabulary `excluded`/`unauthorized`/`unavailable`/`stale`/`archived`/`ambiguous`/`tenant_mismatch`/`conflict`/`invalidReference` reason codes (already enforced as the only output language across all Epic 2 reviews).
- Cross-surface idempotency parity and the safe-denial 404 contract from Stories 1.3/1.4/1.6.

Epic 3 also has a small set of decision points that are already aligned but should be explicit per story:

- Story 3.1 must define the include/exclude decision matrix as a pure policy with a stable evaluation order — Epic 2 has established that authorization (claim → tenant access → project belonging → ACL evidence) is the first gate, then lifecycle, then freshness. The matrix should be written down once.
- Story 3.2 must extend `NoPayloadLeakage` over the assembly DTOs, not just the source events.
- Story 3.4 must explicitly cover the (evidence-state × Refresh) slice of the fail-closed negative-evidence matrix — `Stale` and `Unavailable` are now real producible states in Epic 2's Folders ACL; Memories' fail-closed-clean exclusion is canonical per the ADR.

No Epic 2 discovery fundamentally invalidates the Epic 3 plan. The plan remains sound.
